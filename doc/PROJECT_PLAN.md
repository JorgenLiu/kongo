# Kongo 项目规划

## 产品定位

Kongo 是一个本地优先的关系管理工具，目标不是做一个传统通讯录，而是围绕“联系人 -> 事件 -> 每日总结 -> 附件沉淀”组织个人工作信息。

它当前更像一个桌面工作台：

1. 用联系人承载长期关系背景
2. 用事件承载具体互动与协作记录
3. 用每日总结承载工作推进与次日计划
4. 用附件承载外部文件证据与材料

## 当前实现边界

### 已实现

#### 联系人与标签
- 联系人列表、关键字搜索、创建、编辑、删除、详情页
- 标签创建、编辑、删除
- 联系人打标与标签过滤

#### 事件
- 事件列表与详情页
- 事件创建、编辑、删除
- 多参与人关系维护
- 事件类型管理基础能力
- 关键字搜索与事件类型过滤

#### 每日总结
- 每日总结列表
- 按关键字搜索
- 按日期唯一创建 / 编辑 / 删除
- 从文本中提取行动项

#### 附件与文件库
- 本地文件复制入库
- 关联到事件或每日总结
- 打开、解绑、删除
- 文件库总览与关键字搜索

#### 检索与导航
- 全局检索页
- 命中词高亮
- 联系人 / 事件 / 每日总结结果排序
- 桌面壳导航与分栏结构

### 当前未完全闭环
- 联系人详情中的附件模块仍是轻量入口提示
- 文件库缺少排序、筛选、批量操作与预览增强
- 提醒字段存在，但系统提醒未完成跨平台落地
- 设置页仍偏“项目说明页”，不是真正的用户设置中心

## 当前产品决策

### 1. 事件状态不再是主概念
当前产品已主动移除运行时事件状态概念：

- `Event` 模型不含 `status`
- 表单、详情、列表都不再展示状态
- 日程分组改按时间窗口，而不是按状态

保留项：数据库仍保留 `events.status` 列，作为历史兼容字段。

### 2. 总结以“每日总结”作为当前主模型
当前总结能力不是“事件总结列表”，而是独立的 `DailySummary`：

- 以日期为唯一主线
- 文本结构拆成“当日总结”和“明日计划”
- 行动项从文本中抽取

### 3. 标签与文件是已实现的次级模块
标签管理页和文件库都已实现，但目前不在一级导航中，而是作为次级入口存在。

## 当前版本目标

### 当前阶段目标
把已有数据模型和基础能力进一步补成更完整的使用闭环，而不是继续扩张新的抽象层。

### 当前阶段重点

#### P0：提醒与跨平台系统能力
- 设计并实现 macOS / Windows / iOS 的本地提醒闭环
- 梳理 Windows 发布身份、AUMID / MSIX 等发布前置
- 明确提醒数据模型与系统通知映射关系

#### P1：桌面端体验补强
- 快捷键
- 右键菜单
- Hover 态
- 更完整的设置页与窗口行为

#### P1：文件与联系人模块深化
- 联系人详情中的附件模块升级为真实页面
- 文件库补齐排序、筛选、元数据展示与错误反馈

#### P1：检索继续扩展
- 评估把附件加入全局检索结果
- 继续提升搜索相关跳转与筛选体验

#### P2：平台验证与发布准备
- 补齐 Windows / iOS 真机或完整构建验证
- 梳理平台差异与发布文档

## 非当前优先级

以下方向保留，但不是当前迭代主目标：

1. 云同步
2. 多端实时协作
3. 大规模 AI 工作流自动化
4. 复杂附件解析流水线

## 里程碑建议

### M1：桌面产品基础稳固
- 系统提醒方案确定
- 设置页与桌面交互补齐
- 文件库体验达到可日用水平

### M2：跨平台能力验证
- Windows 构建、身份与通知链路跑通
- iOS 提醒与基础交互验证

### M3：信息沉淀深化
- 联系人详情模块化
- 附件与检索进一步融合

## 事实来源说明

本规划文档以当前代码为准，而不是早期设计设想。若代码与规划冲突，应优先更新规划，而不是沿用过期说法。
