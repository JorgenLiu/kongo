# Kongo 项目关键决策记录

## 📋 关键决策清单

### 1. 技术栈选择

#### 决策: 选择Flutter + Dart + SQLite

**背景**:
- 需要支持macOS、Windows、iOS多平台
- 需要快速开发和维护
- 需要优秀的用户体验

**备选方案**:
1. ❌ 原生开发 (Swift + Kotlin)
   - 优点: 性能最优
   - 缺点: 需要多个开发团队，维护成本高

2. ❌ React Native
   - 优点: JavaScript生态
   - 缺点: macOS和Windows支持较弱

3. ✅ **Flutter** (最终选择)
   - 优点: 一套代码支持多平台、性能好、生态完整
   - 缺点: 学习曲线可能较陡

**决策日期**: 2026年3月6日  
**批准人**: 项目经理  
**影响**: 整个项目的技术基础

---

### 2. 项目平台策略

#### 决策: 从macOS开始，后续扩展iOS和Windows

**理由**:
- macOS是开发者最熟悉的平台
- 可以快速验证产品概念
- 后续扩展Windows和iOS相对容易

**平台顺序**:
1. **Phase 1**: macOS (桌面应用)
2. **Phase 2**: iOS (移动应用)
3. **Phase 3**: Windows (跨平台补完)

**时间规划**:
- macOS: v1.0 (第一版)
- iOS: v2.0 (第二版)
- Windows: v2.1 (第二版补丁)

---

### 3. 数据存储方案

#### 决策: 使用SQLite本地存储，不用云同步

**背景**:
- 第一版需求是本地化应用
- 云同步增加复杂度和成本
- 本地存储满足基本需求

**为什么不用**:
- ❌ Firebase: 增加依赖和成本
- ❌ SQLite + 云同步: 复杂度高
- ❌ Hive/ObjectBox: 功能不必要

**为什么选SQLite**:
- ✅ 文件数据库，便于备份
- ✅ 支持复杂查询
- ✅ 成熟稳定
- ✅ 零成本

**备份和同步计划**:
- v1.0: 本地备份/恢复
- v2.0: 云同步可选功能

---

### 4. 状态管理方案

#### 决策: 使用Provider作为状态管理框架

**备选方案**:
1. ❌ Bloc
   - 优点: 功能完整、清晰的流向
   - 缺点: 代码量多、学习曲线陡

2. ❌ GetX
   - 优点: API简洁、功能全面
   - 缺点: 单一库依赖过多

3. ✅ **Provider** (最终选择)
   - 优点: 官方推荐、轻量级、易学、生产级
   - 缺点: 基础功能，复杂场景需扩展

**理由**:
- 项目规模中等，Provider足够
- 团队易于上手和维护
- 官方推荐，社区支持好
- 性能已验证

---

### 5. 项目架构选择

#### 决策: 采用5层分层架构

**架构层级**:
1. **UI Layer** - 页面和屏幕
2. **Widget Layer** - 可复用UI组件
3. **Provider Layer** - 状态管理
4. **Service Layer** - 业务逻辑
5. **Repository Layer** - 数据访问
6. **Database Layer** - SQLite

**理由**:
- ✅ 清晰的职责分离
- ✅ 易于测试和维护
- ✅ 扩展灵活
- ✅ 团队协作高效

**与其他架构的对比**:
- ❌ MVC: 太简陋，不适合大项目
- ❌ MVP: 测试友好但通常过度设计
- ✅ **MVVM** (Provider): 最适合Flutter应用

---

### 6. 数据库表设计

#### 决策: 5个核心表 + 多对多关联表

**表设计**:
```
contacts ←→ contact_tags ←→ tags
contacts ←→ contact_events ←→ event_types
```

**为什么是5个表而非更多**:
- ✅ 符合数据库规范化原则
- ✅ 避免冗余数据
- ✅ 支持灵活查询
- ✅ 不过度复杂

**为什么不采用单表设计**:
- ❌ 标签和事件如果存在contact表，需要JSON或TEXT字段
- ❌ 不支持有效的索引和查询优化
- ❌ 难以维护数据完整性

---

### 7. 搜索功能设计

#### 决策: 支持全文搜索 + 多标签OR/AND搜索 + 组合搜索

**搜索模式**:
1. **全文搜索**: 在name、phone、email、address、notes中搜索
2. **标签搜索**: 
   - OR模式: 包含任意标签的联系人
   - AND模式: 同时包含所有标签的联系人
3. **组合搜索**: 关键词 + 标签的交集

**为什么选择这些模式**:
- ✅ 覆盖大多数用户搜索场景
- ✅ 用SQL容易实现
- ✅ 性能可控

**不考虑的方案**:
- ❌ 模糊搜索: 实现复杂，收益小
- ❌ 语义搜索: 成本太高
- ❌ 全文索引: 数据量小不必要

### 11. 项目结构组织

#### 决策: Feature-based分包结构

**结构原则**:
```
lib/
├── config/          # 全局配置
├── models/          # 数据模型
├── services/        # 业务逻辑
├── repositories/    # 数据访问
├── providers/       # 状态管理
├── screens/         # 页面（按功能分包）
├── widgets/         # 通用组件
├── utils/           # 工具函数
└── exceptions/      # 异常定义
```

**为什么不用layer-based**:
- Feature-based更易于扩展
- 每个功能独立，合作冲突少
- 便于特性团队工作
