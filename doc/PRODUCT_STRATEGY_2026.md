# Kongo 产品战略与技术路线图

> 撰写时间：2026-03  
> 本文档记录了一次完整的产品战略讨论，涵盖产品定位、AI 接入路线、数据导入策略、云同步方向，以及各项决策的推理过程。  
> 这是当前阶段的事实决策文档，应优先于 `AI_INTEGRATION_ANALYSIS.md` 中的旧分析。

---

## 一、产品定位

### 核心定位

**Kongo 是个人职业人脉的长期资产管理工具——数据归你，关系是你的。**

### 与同类工具的本质区别

| 工具 | 本质 | 根本缺陷 |
|------|------|---------|
| 企业微信 / Salesforce | 公司 CRM | 数据归公司，离职清零；管理员可见所有沟通 |
| 手机通讯录 | 联系方式存储 | 无 context，无互动历史 |
| 微信 | 即时通讯 | 碎片化，无法跨时间沉淀 |
| Excel / Notion | 手动维护 | 无结构，更新全靠自律 |
| LinkedIn | 弱关系展示 | 无法记录深度互动，互动历史不归你管 |

**没有任何现有工具在做"跨越职业生涯的个人关系资产管理"**——这是 Kongo 的核心空白地带。

### 目标用户

主要面向**有主动经营个人人脉需求的个人用户**：

- 销售总监 / BD / 猎头：人脉是核心生产资料，跨越多家公司积累
- 创业者 / 合伙人：每个联系人背后都有数年故事，不能丢
- 有职业规划意识的个人：希望自己的关系网络随时间增值而非贬值

买单人是**个人**，不是公司。这和企业微信不是竞争关系，是两个完全不同的市场。

### "本地优先"的含义

采用**含义一**：数据存储在用户本机，不依赖 Kongo 服务器。允许用户自行授权第三方服务（如 Outlook），数据从第三方直接拉到本地，不经 Kongo 服务器中转。AI 调用使用用户自己的 API key，Kongo 不做 AI 代理。

## 二、核心交互模型：AI 秘书视图

### 产品原点

本产品源自一个明确的需求：**AI 能像私人秘书一样，为用户主动处理人际关系**。

这决定了 Kongo 的核心交互模型不是"用户操作数据库"，而是"秘书在等你"。

### 两种模式的本质区别

| 档案室模式（放弃）| 秘书模式（选定）|
|----------------|--------------|
| 用户打开 app → 看数据 → 找联系人 → 看洞察 | 用户打开 app → 系统已经把今天最值得处理的对象摆到前面 |
| AI 是功能点缀 | 提醒系统是主入口，AI 是增强层 |
| 用户拉取信息 | 系统主动提示，用户顺手沉淀 |

### 首页形态：工作台 + 单张优先提醒卡

首页回归"工作台"定位，不再让 AI 简报成为第一主角。首页第一职责仍然是回答：今天有什么安排、这一周什么节奏、有没有临近待处理事项。

在工作台之上，只增加**一张优先提醒卡**，用于提示今天最值得顺手补充上下文的会议或联系人：

```
──────────────────────────────
  今日提醒（3月26日 星期四）
──────────────────────────────
  [会前准备] 15:00 产品复盘会
  参会人 Alex Foster 还没有背景信息
      → 补一句备注  → 标记这个人是谁
──────────────────────────────
```

第一版不做 3 到 6 条秘书卡片流，也不要求 AI 生成首页摘要。提醒卡由确定性规则生成，强调三件事：

- 为什么今天会出现这条提醒
- 用户此刻可以立刻做什么
- 做完后系统会多记住一点什么

AI 仍然保留，但从首页主入口后撤。它更适合在数据成熟后处理非结构化内容，例如会议纪要整理、联系人关系上下文回顾、跟进文案草拟与更深层洞察。对话式助手仍为长期演进方向，不在当前迭代实现。

### 主动推送架构预留

秘书体验的核心仍然是**主动**，不是被动等用户打开 app。但近期应以确定性提醒为主，而不是首页 AI 简报。当前阶段需同步预留：

- 定时任务触发器（每日生成提醒，不依赖用户打开 app）
- 系统通知推送（macOS / Windows 桌面通知）
- 提醒去重与已处理状态，避免重复打扰

## 三、AI 接入路线

### 现有基础设施（已完成，无需重建）

以下内容已在代码库中全部实现，与旧版 `AI_INTEGRATION_ANALYSIS.md` 的"缺失"列表不符：

- `AiProvider` 抽象接口（`lib/ai/ai_provider.dart`）
- `AiService` + `DefaultAiService` 编排层（`lib/ai/ai_service.dart`）
- `MockAiProvider` 测试双（`lib/ai/mock_ai_provider.dart`）
- `AiJob` / `AiOutput` 模型（`lib/models/ai_job.dart`）
- `SqliteAiJobRepository` 持久化层（`lib/repositories/ai_job_repository.dart`）
- `ai_jobs` / `ai_outputs` 数据库表（已在 schema 中）
- `AppDependencies` 中的 `DefaultAiService` 布线（但 `provider: null`，即 `isAvailable = false`）
- `Provider<AiService>` 已在 widget tree 中

**唯一阻塞实际 AI 调用的缺口**：没有真实的 HTTP provider 实现 + 没有设置页让用户填写 API key。

### AI Provider 策略：预设列表 + 自定义

设置页提供预设 provider 选项，用户选择后只需填对应 API key，baseUrl 自动填充：

| Provider | baseUrl | 默认模型 |
|----------|---------|---------|
| 硅基流动 | `https://api.siliconflow.cn/v1` | `deepseek-ai/DeepSeek-V3` |
| DeepSeek 官方 | `https://api.deepseek.com/v1` | `deepseek-chat` |
| 通义千问（DashScope） | `https://dashscope.aliyuncs.com/compatible-mode/v1` | `qwen-plus` |
| 自定义 | 用户填写 | 用户填写 |

所有 provider 均为 OpenAI 兼容格式，共用同一套 HTTP 实现（`OpenAiCompatibleProvider`），只是初始化参数不同。这是单一实现，不需要每个 provider 单独维护一个类。

### AI 功能接入优先级

#### Phase 1：AI 基础设施通路（已完成）

以下内容已在 2026-03 期间全部落地：

- `OpenAiCompatibleProvider` 实现 `AiProvider` 接口，支持硅基流动、DeepSeek、通义千问、自定义端点
- 设置页"AI 能力"配置区块：provider 选择 + API key 输入 + 测试连接
- API key 通过 macOS Keychain / iOS Keychain 存储（Swift 桥接）
- `AppDependencies.bootstrap()` 启动时按配置注入 `AiService`

基础设施底座完整，无需重建。

#### Phase 2（暂停）：首页智能日报

首页 AI 日报 V1 已技术落地，但经过 2026-03-28 评估后**不再推进**：冷启动阶段无足够上下文时，AI 只能将结构化字段翻译为自然语言，无增量信息量。首页主入口改为规则驱动的单张提醒卡（确定性逻辑，无 AI 调用）。

#### Phase 3（中期）：联系人关系洞察

**场景**：联系人详情页，AI 根据该联系人的事件历史、备注、标签、里程碑，生成跟进提示和关系摘要。触发时机：用户已积累 5+ 条关于该联系人的 quick_note 或事件记录后，用户主动点击"生成摘要"。

**呈现位置**：详情页顶部折叠卡片，默认收起，标题显示"AI 洞察 · 数据截至 YYYY-MM-DD"，点击展开。

**持久化**：AI 输出存入 `ai_outputs`（outputType=`contact_insight`，targetId=contactId）。有新事件后标记缓存失效，用户手动点"刷新"重新生成。

**约束**：所有 AI 调用必须用户主动触发，不允许后台自动发送。

#### Phase 4（中期）：事件"会后补充"

**场景**：事件结束后，事件详情页顶部出现提示横幅，附带单行快速备注输入框。用户一句话，AI 帮结构化为备注 + 可能的行动项。

**设计原则**：把"会后记录"从"创建新内容"降级为"补全已有骨架"，摩擦大幅降低。

## 三、数据入口策略

### 核心问题

用户的关系数据散落在微信、Outlook、手机通讯录，没有任何工具帮助他们把这些整合到一个地方。但对管理者 / 高级职场人来说，真正的阻力并不是"没有导入入口"，而是**他们不愿意把导入和维护当成一项显式任务来完成**。

因此，Kongo 的 onboarding 成败取决于两件事是否同时成立：

1. 用户第一次打开 app，能不能立刻看懂一条真实有用的提醒
2. 数据能不能在低摩擦的前提下自然进入系统，而不是要求用户先坐下来整理 CRM

### 数据入口优先级

#### 入口一：Microsoft Outlook + Teams 日历（最高价值的数据加速器）

Teams 会议本质上是 Outlook 日历事件，接入 Microsoft Graph 即同时获得两者。

- **授权方式**：Azure AD 注册应用 → OAuth 2.0 → `flutter_web_auth_2` 包处理本地回调（localhost 监听或 URL scheme）
- **数据来源**：`/me/calendarView`（含 Teams 会议）+ `/me/events/{id}/attendees`（参与人 → 联系人池）+ `/me/contacts`（Outlook 通讯录）
- **权限 scope**：`Calendars.Read Contacts.Read User.Read offline_access`（仅读取，无写操作）
- **关键价值**：会议 attendees 是企业用户真实互动过的人，比任何手动录入都准确；连接后可显著减少重复建会与补录参会人的劳动
- **跨平台**：macOS + Windows 均可，纯 HTTP + OAuth，无原生代码

但这一路径不是 onboarding 的唯一前置条件。对于当前目标用户，Outlook / Teams 更适合作为**在用户理解价值之后出现的数据加速器**，而不是第一次打开产品时必须完成的授权步骤。

#### 入口二：vCard / CSV 文件导入（保底通用）

- iPhone 通讯录可导出 `.vcf`，Outlook 可导出 `.csv`
- 纯 Dart 解析，实现成本一天内可完成
- LinkedIn 导出（设置 → 数据隐私 → 获取数据副本 → `Connections.csv`）是高质量商业人脉来源

#### 入口三：文件拖拽 + AI 解析（兜底所有漏网信息）

用户把任何文本/截图粘贴或拖入 Kongo，AI 提取联系人/事件/待办，用户确认后写入。

**关键原则：AI 提取后必须经用户确认，不能自动写入**——一次识别错误会污染联系人库。

`desktop_drop` 包支持 macOS + Windows + Linux，无原生代码。

#### 入口四：企业微信通讯录（B 端定向功能，后期）

- 个人用户无法导出企业微信通讯录（平台刻意封闭）
- 企业管理员可通过企业微信 API 获取通讯录
- 适合作为 B 端付费集成功能，不作为标准功能

#### 微信个人联系人：彻底排除（最终决策）

经过完整调研和论证，所有微信数据集成路线均已排除：

| 路线 | 排除理由 |
|------|---------|
| 微信 OAuth 授权登录 | 只能获取用户自身 openid/昵称/头像，**无法获取好友列表**；社交关系图谱 API 仅对大型机构合作伙伴开放，独立开发者申请不到 |
| 微信服务号（官方 API）| 技术合规，但需要云端服务器接收 webhook，**与本地优先原则根本冲突**；用户仍需手动转发，摩擦降低有限 |
| 非官方 bot 框架（itchat 等）| 违反微信 ToS，腾讯主动封号，不适合做成产品 |
| 解密本地数据库（EnMicroMsg.db）| 需要 root/越狱，普通用户无法操作 |

**腾讯通过龙虾（微信 AI 助手）守住了这个生态，外部无法合法复制。**

微信数据的唯一可行补充：用户把聊天截图手动粘贴到菜单栏浮窗，AI 帮提取关键信息——这是"入口三：文件拖拽 + AI 解析"的自然覆盖范围，无需单独实现。

#### 引导式 Onboarding（首次启动）

用户装完 Kongo 面对空列表会直接关掉。需要一个引导流程解决冷启动问题，但主路径不应是"先导入"，而应是"先理解价值，再开始沉淀，再顺势连接数据源"：

```
第一步：空首页 → 主 CTA："先看一个示例提醒"
第二步：示例提醒页 → 展示一张会议前准备卡，让用户理解 Kongo 如何围绕会议与联系人产生提醒
第三步：给出两个真实入口
  ├── 从今天的一场会议开始
  └── 手动添加一个联系人
第四步：用户完成第一次轻量沉淀
  ├── 补一句备注
  ├── 标记这个人是谁
  └── 设置会后提醒
第五步：在第一次沉淀完成后，再提示连接 Outlook / Teams
  └── 诉求不是"解锁完整功能"，而是"以后不用再手动建会议"
```

这个流程的目标不是让用户"完成导入"，而是让用户在第一次会话内完成一个闭环：

1. 看懂 Kongo 是干什么的
2. 从一个真实对象开始使用
3. 完成一次极轻的沉淀动作
4. 感受到连接 Outlook / Teams 可以减少重复劳动

文件导入与 AI 解析仍然保留为补充入口，但不应承担主路径职责。

#### 文档知识库：只做关系型检索，不做通用知识库（当前决策）

用户访谈中提到的"知识库"需求真实存在，但当前阶段不应把 Kongo 扩张成通用文档管理产品。Kongo 只处理**与联系人、事件、项目上下文强相关的文档**，例如合同、报价单、会议纪要、方案、截图、往来材料。

这意味着 Kongo 当前做的是**关系型文档层**，不是 Notion/Obsidian/NotebookLM 式的通用知识库。

#### 文档知识库 v1：本地解析 + 全局全文检索

第一阶段只做完全本地、零额外推理成本的方案：

1. 用户上传文档后，优先在本地提取文本
2. 提取结果写入 SQLite FTS5 全文索引
3. 文档继续作为附件挂在联系人/事件名下
4. 全局搜索同时命中联系人、事件、文档内容

目标不是让 AI "理解整个知识库"，而是先解决用户最现实的问题：**找文件时找不到。**

建议的数据形态如下：

```sql
CREATE VIRTUAL TABLE attachment_fts USING fts5(
  attachment_id UNINDEXED,
  contact_id UNINDEXED,
  event_id UNINDEXED,
  content,
  tokenize = 'unicode61'
);
```

这个方案的优点：
- 完全本地，符合本地优先原则
- 不依赖向量数据库
- 不依赖常驻 AI 成本
- 可以先解决 PDF/文本文档/截图 OCR 后文本的统一检索问题

#### 文档知识库 v2：上传后异步生成 AI 摘要（延后决策）

第二阶段再考虑在文档上传完成后，调用用户自带 API key 的模型生成摘要、关键词、可能关联的人/事件，再把这些派生结果一并写入检索索引。

但在 v1 阶段，**不引入完整 RAG，不做 embedding，不做向量检索**。原因很明确：
- 需要额外存储和索引体系
- 上传成本和维护复杂度明显上升
- 当前用户核心痛点首先是"找得到"，不是"跨文档推理"

#### 是否保留每次 AI 对知识库的更新：当前结论

当前不应把 AI 输出当成需要不断累积版本的"知识库主数据"。主数据始终应是：
- 原始文档文件
- 本地提取出的原文文本
- 文档与联系人/事件的关联关系

AI 生成的摘要、关键词、推荐关联，都是**可重建的派生数据**。未来如果引入 v2，应优先保存：
- 原始文档 hash
- 最近一次解析时间
- 最近一次 AI 摘要结果
- 最近一次摘要对应的模型/version

而不是保存每一次 AI 改写历史。除非后续明确出现"审计"或"人工回滚摘要"需求，否则没有必要为派生层建立复杂版本系统。

## 四、用户行为设计：如何驱动持续记录

### 根本问题：习惯惯性

单靠功能无法改变用户行为。**用户对已有工作流有强烈惯性，要求他们接受新的交互成本是最核心的产品挑战。**

用户研究（企业高管访谈）验证了以下核心痛点：
- **会议承诺遗漏**：不是缺少任务管理工具，而是"这次开会我同意了一个价格，执行时忘了"——会议中产生的决定/承诺没有被捕获
- **见的人记不住**：见过的人、谈过什么、当时什么背景，随时间蒸发
- 用户已在用日历，日程不会遗漏；遗漏的是**会议里产生的上下文**

用户表示愿意接受与 app 的交互成本——**前提是数据入口要尽可能自动化**，否则又回到"靠自律"的死局。

### 核心设计原则：触发器是会议，不是自律

**不要求用户改变习惯，而是把 Kongo 附着在已有习惯上。**

用户的已有习惯是：开会、接电话、谈客户。Kongo 应该在这些时刻的前后主动出现，而不是等用户想起来打开 app。

| 已有习惯 | Kongo 的介入时机 | 用户操作成本 |
|---------|---------------|----------|
| 看到明天的日历 | 会前30分钟推送准备卡 | 零（被动接收） |
| 会议刚结束 | 系统通知"有什么要记录的" | 3秒（响应通知，一句话） |
| 随时遇到想记录的事 | 菜单栏 `⌘⇧K` 唤起浮窗 | 5秒（不需要切换 app） |
| 每周一早上 | 推送本周待跟进清单 | 零（被动接收） |

**记录的触发器是会议和时刻，不是用户的记性和自律。**

### 解法：会前价值倒逼会后记录

这是行为科学中的"完成循环"效应——如果 Kongo 在**会议开始前**就给了用户价值（如联系人洞察显示"你们上次讨论了 X，距今 40 天"），用户在会议里占到了便宜，会有强烈的心理冲动更新这条信息。

**联系人洞察和首页日报不只是 AI 功能，它们是催生"会后记录"习惯的钩子。Kongo 先给，用户才想还。**

### 行为闭环

```
会前：首页日报 / 联系人洞察 → 用户在会议中获得价值
会中：事件已建，骨架已在（参与人自动从日历同步）
会后：系统通知 / 菜单栏浮窗 → 一句话快速捕获（包括微信口头承诺）
次日：首页日报回显昨天的记录和今日建议 → 正向强化
```

每个 AI 接入点都在加固这个行为闭环。

### 菜单栏捕获 vs Outlook 集成：互补而非竞争

这两个入口覆盖不同场景，需要同时存在：

| | Outlook 日历集成 | 菜单栏快速捕获 |
|---|---|---|
| 解决什么 | 正式会议数据自动进来 | 微信/电话/口头承诺能进来 |
| 用户行为 | 零操作（完全被动） | 5秒操作（主动但极低摩擦） |
| 覆盖场景 | Teams 会议、Outlook 日程 | 一切非正式沟通和临时承诺 |

## 五、云同步决策

### 决策

**确认引入云同步，支持 PC 端与移动端数据同步。** 这是本次讨论的明确结论。

### 同步方案路线图

| 用户群 | 同步方案 | 实现时机 |
|--------|---------|---------|
| macOS + iOS（Apple 用户） | iCloud CloudKit（record-level 同步，非文件同步）| 长期 Phase |
| 跨平台（含 Windows） | E2EE 中继（客户端本地加密后上传 blob，服务器不见明文）| 延后，另起计划 |
| 不需要同步 | 纯本地，完全不受影响 | 始终支持 |

> **注意**：不使用将 SQLite 文件放在 iCloud Drive / OneDrive 中直接同步的方式——多设备并发写入会导致数据库损坏。必须使用 record-level 的 CloudKit API。

### 立即要做的一件事：Schema 预埋

云同步与否，现在就需要在各核心表补充以下字段，避免日后大规模迁移：

```sql
-- 在 contacts / events / daily_summaries / tags / attachments 等核心表添加：
updated_at  INTEGER NOT NULL DEFAULT (strftime('%s','now') * 1000)  -- Unix 毫秒
deleted_at  INTEGER  -- NULL = 存活，有值 = 软删除
```

**`updated_at`**：冲突解决的基础，最后写入者胜（last-write-wins）。  
**`deleted_at`**：软删除，防止"你删我补"的同步循环。现有的硬删除查询需要增加 `WHERE deleted_at IS NULL` 过滤。

这两个字段对当前功能零影响，现在处理成本最低。

## 六、跨平台边界确认

所有功能均在 macOS + Windows 成立，无需维护两套逻辑：

| 功能 | macOS | Windows | 实现方式 |
|------|-------|---------|---------|
| 菜单栏 Quick Capture | ✅ | ⚠️ 另起计划 | Swift NSStatusItem + MethodChannel（macOS）；Windows 用 tray_manager（延后）|
| AI HTTP 调用 | ✅ | ✅ | `http` 包，纯 Dart |
| API Key 安全存储 | ✅ Keychain | ✅ Credential Store | Swift 桥接（macOS/iOS）；Windows 用 flutter_secure_storage |
| Outlook 日历 / 通讯录 | ✅ | ✅ | Microsoft Graph HTTP + OAuth |
| vCard / CSV 导入 | ✅ | ✅ | 纯 Dart 文件解析 |
| 文件拖拽解析 | ✅ | ✅ | `desktop_drop` |
| iCloud CloudKit 同步 | ✅ | ❌ 不支持 | Apple 生态专属，Windows 用 E2EE 中继替代 |
| iOS 主屏幕 Widget | ✅ flutter_home_widget | ❌ | 依赖 CloudKit 同步先落地 |

**macOS 菜单栏实现说明**：不使用第三方包（`tray_manager`），采用 Swift 原生 NSStatusItem + NSPopover + NSEvent 全局快捷键，通过 MethodChannel 与 Flutter 通信。原因见 `TECH_STACK_DECISIONS_2026_03_28.md`。

## 七、实施路线图

### 近期（当前迭代）

1. **首页复原** — 移除 AI 日报卡，恢复工作台布局；AI 基础设施保留

13. **文档知识库 v2** — 上传后异步 AI 摘要作为检索增强层

## 八、关键决策索引

| 决策 | 结论 |
|------|------|
| 核心交互模型 | 秘书模式：提醒系统是主入口（规则驱动），AI 是数据成熟后的增强层 |
| 首页形态 | 工作台 + 单张规则驱动提醒卡；AI 日报从首页主路径上撤（2026-03-28 评估后确认）|
| 首页 AI 日报 | 技术已落地，策略上不再推进；冷启动无上下文时 AI 无增量价值 |
| AI provider 形式 | 预设列表 + 自定义，用户填 API key |
| 首选 AI provider | 硅基流动（DeepSeek 模型）+ 同时支持 DeepSeek 官方 / 通义千问 / 自定义 |
| AI provider 实现 | 单一 `OpenAiCompatibleProvider`，所有 provider 共用 |
| 菜单栏 Quick Capture | **当前 P0 主线**；Swift 原生 NSStatusItem + MethodChannel；无 AI，全程离线解析 |
| 菜单栏快捷键 | `⌃⌘K`（原方案 `⌘⇧K` 调整，避免与系统冲突）|
| 云同步方案 | macOS+iOS 用 iCloud CloudKit（record-level，非文件同步）；Windows 用 E2EE 中继（延后）|
| 桌面小组件（iOS Widget）| **保留**，依赖 CloudKit 同步先落地；使用 flutter_home_widget；Tauri 不支持（已评估）|
| 日历接入 | Outlook + Teams（Microsoft Graph），优先于 Google Calendar |
| 通讯录接入 | Outlook 通讯录（随日历 OAuth 免费带来）+ LinkedIn CSV + vCard |
| 企业微信 | 排除个人用户路线；企业管理员 API 作为后期 B 端付费功能 |
| 微信个人联系人集成 | **彻底排除**：OAuth 无法获取好友列表，服务号与本地优先冲突，龙虾是腾讯护城河；唯一补充为用户手动截图粘贴到 AI 解析入口 |
| 文档知识库定位 | 只做与联系人/事件强相关的关系型文档层，不做通用知识库产品 |
| 文档知识库 v1 | 本地文本提取 + SQLite FTS5 全文检索，先解决"找得到" |
| 文档知识库 v2 | 上传后异步 AI 摘要作为增强层；不做完整 RAG/向量检索 |
| AI 对知识库的更新 | AI 输出是可重建派生数据，不保留每次改写历史；保留原文、hash、最近一次摘要结果即可 |
| 桌面小组件 | 排除（无法用 Flutter 实现，无跨平台方案） |
| 竞争定位 | 不与企业微信竞争；目标是个人职业人脉长期资产管理 |
| 用户行为核心挑战 | 习惯惯性是第一障碍；解法是把 Kongo 附着在已有习惯（会议）上，而非要求用户建立新习惯 |
| 会后快速记录优先级 | 中期；定性为**核心功能**（直接解决"会议承诺遗漏"痛点），而非体验优化项 |
| 菜单栏快捷捕获优先级 | **当前 P0，近期迭代核心**；Quick Capture 是产品核心交互简语；不依赖 AI |
